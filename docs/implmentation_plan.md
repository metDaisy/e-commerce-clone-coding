# Amazon Clone: 모듈러 모놀리스 & 분산 트랜잭션 패턴 적용

외부 인프라를 최소화(PostgreSQL 단독 사용)하면서도, 향후 MSA로 쉽게 전환할 수 있는 **모듈러 모놀리스(Modular Monolith)** 아키텍처 기반의 14일 집중 프로젝트 계획입니다.

## 🎯 핵심 목표
단일 스프링부트 애플리케이션 내에서 도메인(주문, 결제, 재고) 간의 결합도를 낮추고, **메시지 큐(Kafka)가 있는 것처럼 가정하여 이벤트 기반 아키텍처와 Outbox 패턴**을 연습합니다.

1. **도메인 결합도 최소화**: 모듈러 모놀리스 구조 설계
2. **이벤트 기반 트랜잭션 (Saga & Outbox 적용)**: 외부 메시지 큐 없이 스프링 내부 이벤트(`ApplicationEvent`)로 메시징 시스템 시뮬레이션
3. **단일 인스턴스 기반 실시간 웹소켓 통신**

---

## 🏗 아키텍처 및 기술 스택

### 1. 모듈러 모놀리스 (Modular Monolith) 구조
하나의 프로젝트 안에 폴더(또는 Gradle 멀티 모듈) 단위로 서비스를 철저히 분리합니다. 서로 다른 도메인은 직접적인 메서드 호출을 최소화하고 '이벤트'를 통해 소통합니다.
- **Order Module (주문)**
- **Payment Module (결제)**
- **Inventory Module (재고)**

### 2. 기술 스택 (외부 인프라 최소화)
- **Backend**: Spring Boot 3.x, Spring Web MVC, Spring Data JPA
- **Database**: PostgreSQL (단일 DB 내에서 스키마 분리 또는 테이블 접두사로 논리적 분리)
- **Messaging (Simulated)**: Spring `ApplicationEventPublisher` 및 `@Scheduled`
- **Real-time**: Spring WebSocket (단일 인스턴스 인메모리 세션 관리)

---

## 🚀 주요 구현 포인트

### 1. Outbox 패턴 & 이벤트 퍼블리싱 (메시지 큐 대체)
- 주문이 발생하면 `orders` 테이블에 저장함과 동시에, 같은 트랜잭션으로 `outbox` 테이블에 '주문 생성 이벤트'를 저장합니다.
- 스프링 `@Scheduled` 스케줄러가 주기적으로(예: 1~2초) `outbox` 테이블을 폴링(Polling)하여 아직 발행되지 않은 이벤트를 읽어옵니다.
- 읽어온 이벤트를 스프링의 `ApplicationEvent`로 발행하고, 발행 완료 처리(상태 업데이트)를 합니다.
- 결제/재고 모듈은 이 내부 이벤트를 리스닝하여 비동기(`@Async`)로 작업을 수행합니다. (나중에 스케줄러 부분만 Kafka Producer로 교체하면 완벽한 분산 시스템이 됩니다.)

### 2. 단일 인스턴스 웹소켓: 타임세일 실시간 재고 갱신
- 사용자들이 상품 상세 페이지에 접속하면 웹소켓에 연결됩니다.
- 재고 모듈에서 재고가 성공적으로 차감되면 이벤트를 발생시키고, 웹소켓 핸들러가 현재 연결된 모든 사용자에게 남은 재고 수량을 브로드캐스팅합니다.
- 단일 인스턴스이므로 Redis Pub/Sub 없이 스프링 내부의 세션 메모리만으로 충분히 동작합니다.
